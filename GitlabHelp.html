Configurer un corpus GitLab pour SonalPi

Cette page explique les étapes à réaliser une seule fois avant de pouvoir connecter SonalPi à un projet GitLab. Ces opérations se font directement dans l'interface web de votre instance GitLab universitaire.

⚙️ Étapes de configuration

1
Créer un groupe GitLab pour votre équipe Optionnel

Sur certaines instances GitLab universitaires, l'option New project n'apparaît que lorsque vous êtes à l'intérieur d'un groupe. Si c'est le cas chez vous, commencez par créer un groupe. Sinon, passez directement à l'étape 2.

L'autre avantage d'un groupe : vous gérez les membres une seule fois et ils ont accès à tous les projets du groupe.

  • Connectez-vous à votre instance GitLab : https://gitlab.univ-exemple.fr
  • Dans le menu principal (barre de gauche ou menu "+" en haut), choisissez New group → Create group
  • Donnez-lui un nom (ex. : labo-linguistique) et choisissez la visibilité Private
  • Invitez vos collègues via Manage → Members → Invite members, avec le rôle Developer au minimum
ℹ️ Si un groupe existe déjà pour votre équipe, utilisez-le directement à l'étape 2.
Si vous ne créez pas de groupe, vous pourrez inviter vos collaborateurs directement sur le projet à l'étape 2.
2
Créer le projet GitLab (dépôt du corpus)

Le corpus sera stocké dans un dépôt Git privé. La procédure diffère selon que vous avez créé un groupe à l'étape 1 ou non.

Avec un groupe (étape 1 effectuée)
  • Naviguez dans votre groupe, puis cliquez sur New project → Create blank project
  • Choisissez la visibilité Private
  • Le chemin du projet sera : groupe/nom-projet
    (exemple : labo-linguistique/corpus-oral-2024)
  • Tous les membres du groupe auront automatiquement accès au projet
Sans groupe (étape 1 ignorée)
  • Depuis votre page d'accueil ou le menu "+" en haut, cliquez sur New project → Create blank project
  • Choisissez la visibilité Private
  • Le chemin du projet sera : votre-pseudo/nom-projet
    (exemple : jdupont/corpus-oral-2024)
  • Pour donner accès à vos collègues : Settings → Members → Invite members, avec le rôle Developer au minimum
⚠️ Le projet doit impérativement rester en visibilité Private. Un projet public rendrait les données du corpus accessibles à tous sans authentification, ce qui serait incompatible avec le RGPD si le corpus contient des données personnelles.
3
Activer Git LFS sur le projet

SonalPi utilise les verrous exclusifs LFS de GitLab pour empêcher deux personnes de modifier le même entretien simultanément. Ces verrous s'appliquent sur toutes les branches (et non seulement sur la branche par défaut).

Pour les activer, LFS doit être activé sur le projet :

  • Dans le projet GitLab, allez dans Settings → General
  • Dépliez la section Visibility, project features, permissions
  • Activez le toggle Git Large File Storage (LFS)
  • Cliquez sur Save changes
ℹ️ La configuration du fichier .gitattributes (nécessaire pour déclarer les types de fichiers soumis aux verrous LFS) est gérée automatiquement par SonalPi à la première connexion au projet. Vous n'avez rien à faire manuellement.
⚠️ Si vous n'avez pas les droits pour modifier ces paramètres, demandez à l'administrateur du projet (rôle Maintainer ou Owner).
4
Créer une application OAuth dans votre profil GitLab

SonalPi utilise OAuth 2.0 pour s'authentifier en votre nom sur GitLab. Chaque utilisateur doit créer sa propre application OAuth.

  • Allez dans Préférences utilisateur (icône de profil en haut à droite → Edit profile)
  • Dans le menu de gauche, cliquez sur Applications
  • Cliquez sur Add new application

Remplissez le formulaire ainsi :

  • Name : SonalPi (nom libre)
  • Redirect URI : http://127.0.0.1:7474/callback
  • Confidential : décocher (application non-confidentielle)
  • Scopes : cocher uniquement api

Cliquez sur Save application.

Client ID : après la création, GitLab affiche un Application ID (longue chaîne de caractères). C'est la valeur à copier dans le champ Client ID de SonalPi.

⚠️ Le Secret affiché n'est pas nécessaire pour SonalPi, vous pouvez l'ignorer.
ℹ️ Cette application OAuth reste dans votre profil personnel. Elle ne donne accès qu'aux projets auxquels vous êtes déjà membre.
5
Placer le fichier corpus dans le dépôt

Le fichier .crp (corpus SonalPi) doit être présent dans le dépôt GitLab avant de pouvoir l'ouvrir depuis SonalPi.

  • Si vous démarrez un nouveau corpus : créez d'abord le corpus localement dans SonalPi, puis déposez le fichier .crp et les entretiens .sonal dans le dépôt GitLab via l'interface web (+ → Upload file) ou via Git
  • Si le corpus existe déjà dans le dépôt : notez son chemin depuis la racine, par exemple corpus/mon-projet.crp
⚠️ Les fichiers audio ne sont pas synchronisés via GitLab — ils restent locaux sur chaque machine.

🔚 En fin de projet

6
Récupérer tous les fichiers et supprimer le projet GitLab

Avant de clôturer le projet GitLab, assurez-vous que chaque membre de l'équipe dispose bien de sa copie locale complète du corpus.

1. Vérifier que tout le monde a récupéré les fichiers

  • Chaque collaborateur doit ouvrir SonalPi et effectuer une dernière synchronisation pour rapatrier toute modification encore présente sur GitLab
  • Vérifiez dans GitLab (Repository → Files) que tous les fichiers .crp et .sonal sont bien présents et à jour dans le dépôt
  • Assurez-vous qu'aucun verrou LFS n'est encore actif : Repository → Locked files doit être vide

2. Supprimer le projet GitLab

  • Dans le projet, allez dans Settings → General
  • Dépliez la section Advanced tout en bas
  • Cliquez sur Delete project et confirmez en tapant le nom du projet
Si vous aviez créé un groupe (étape 1 effectuée)
Une fois tous les projets du groupe supprimés, vous pouvez également supprimer le groupe : dans le groupe, Settings → General → Advanced → Remove group.
⚠️ La suppression d'un projet GitLab est irréversible. Vérifiez bien que chaque membre a récupéré sa copie locale avant de procéder. Les fichiers audio locaux ne sont pas concernés (ils n'ont jamais transité par GitLab).

❓ Questions fréquentes

Dois-je répéter ces étapes pour chaque collègue ?
Une seule fois par le responsable du projet : étape 1 si nécessaire (créer le groupe — optionnel), étapes 2 (créer le projet), 3 (activer LFS) et 5 (déposer le corpus).

Une fois par utilisateur : étape 4 uniquement (créer son application OAuth dans son propre profil GitLab). Chaque membre de l'équipe doit créer sa propre application — c'est ce qui permet à GitLab de distinguer qui fait quelle modification dans l'historique.
Le token OAuth expire-t-il ?
Oui. GitLab émet des tokens avec une durée de vie limitée (généralement 2 heures). SonalPi gère le rafraîchissement automatiquement grâce au refresh token. Si la reconnexion échoue, il faudra relancer le flux d'authentification depuis le menu.
Que se passe-t-il si un entretien est verrouillé par un collègue ?
SonalPi affichera un avertissement et ouvrira l'entretien en lecture seule. Le verrou est libéré automatiquement quand le collègue ferme la fenêtre d'édition. Contrairement au mode serveur PHP, les verrous GitLab LFS ne n'expirent pas : si l'application se ferme anormalement, le verrou doit être levé manuellement dans GitLab (Repository → Locked files) ou par un Maintainer du projet.
Pourquoi l'option "Confidentielle" doit-elle être décochée ?
Une application confidentielle requiert qu'un client secret soit envoyé lors de l'échange du token OAuth. Or SonalPi est une application de bureau : il n'existe pas de serveur backend pour conserver ce secret de façon sécurisée. En décochant "Confidential", GitLab accepte le flux sans secret côté client — c'est le comportement standard pour toutes les applications de bureau (VS Code, GitHub Desktop, etc.).

Cela ne compromet pas la confidentialité de vos données. La sécurité du flux est assurée par PKCE (Proof Key for Code Exchange), un mécanisme cryptographique intégré à SonalPi qui remplace avantageusement le client secret pour les applications de bureau.

Du point de vue RGPD, ce qui compte est que le projet GitLab reste en visibilité Private (voir étape 2). C'est cette configuration qui garantit que les données du corpus ne sont accessibles qu'aux membres autorisés de votre équipe — indépendamment du mécanisme d'authentification OAuth utilisé.
Mon instance GitLab utilise un certificat SSL auto-signé, est-ce un problème ?
Non. SonalPi est configuré pour accepter les certificats auto-signés, ce qui est courant sur les instances universitaires en intranet.
